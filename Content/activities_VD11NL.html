ruby:
每个缩排层级使用两个空格，不要使用制表符：个人常用制表符号
(、[ 之后，]、) 之前，不要添加任何空格，在 { 前后，在 } 之前添加空格：括号和中括号需要包紧，而大括号不需要
在各个方法定义之间添加空行，并且将方法分成若干合乎逻辑的段落：不能在方法间添加两个空行
在各个段落之间，使用一个空行分隔
在属性修饰器之后，使用一个空行分隔：书写存取器之后需要空格
在不同缩进的代码之间，不要使用空行分隔：已经有缩进区分开就不在需要空格了
当方法调用参数过长时，将它们排列在多行并对齐：这种情况经常碰见，以前不知道该如何处理，尤其是参数的缩进
永远不要使用 for，除非你很清楚为什么：for是由each实现的，所以尽量使用迭代器。
永远不要使用 and 与 or 关键字。使用 && 与 || 来替代
优先考虑简短的自我赋值语法。

倾向使用 map 而不是 collect，find 而不是 detect，select 而不是 find_all，reduce 而不是 inject 以及 size 而不是 length。这不是一个硬性要求，如果使用别名可以增强可读性，使用它也没关系。这些别名方法继承自 Smalltalk 语言，但在别的语言并不通用。鼓励使用 select 而不是 find_all 的理由是前者与 reject 搭配起来一目了然。


rails
控制器：
控制器应该保持苗条―它们应该只为视图层提供数据，不应包含任何业务逻辑（所有业务逻辑都应当放在模型里）
每个控制器的动作（理论上）应当只调用一个除了初始的 find 或 new 之外的方法。
视图：
不要直接从视图调用模型层。
应使用 partial 模版与布局来减少代码重复
复杂的格式化不应放在视图中，而应提取为视图 helper 或模型中的方法

几个 Ruby 风格指南转码风格技巧：
  1. 定义方法时，避免单行写法；
  2. 操作符前后适当添加空格，在逗号(,)、冒号(:)及分号(;)之后；特别注意在指数操作符后面不加空格，如：'a**2'；
  3. ( 、[ 之后，] 、) 之前，不添加任何空格。在 { 前后，在 } 之前添加空格。需注意 { 与 }，对于哈希字面量，{ 之后 与 } 之前有无空格都可以，保持一致性就好，在Ruby社区用空格更流行；对于插值表达式，括号内”#{user.last_name}"两端不要添加空格;
  4. ! 之后，不要添加任何空格；
  5. 在属性修饰器之后，使用一个空行分隔。
几个Rails 风格指南转码风格技巧：
  1. 将所有环境下都通用的配置放在 config/application.rb 文件中；
  2. 当需要为一个RESTful资源添加动作时，应使用member路由和collection路由；
  3. 控制器应该保持苗条－他们应该只为试图层提供数据，不应包含任何业务逻辑；
  4. 模型的命名应有意义（但简短）且不含缩写；
  5. 自定义验证规则应放在app/validators目录下；
  6. 不要在查询中使用字符串插值，它会使你的代码有被SQL注入攻击的风险；
  7. 应使用版本控制工具记录 schema.rb (或 structure.sql)的变化；
  8. 应使用partial模板与布局涞减少代码重复；
  9. 有缺陷的gem，rmagick(minimagick替代)、autotest(guard/watchr替代)、rock(SimpleCov替代)、therubyracer(node.js替代)。

几个主要的《 Ruby 风格指南》的影响：
  • 每个缩排层级使用两个空格。不要使用制表符。
  • 不要使用 ; 隔开语句与表达式。推论：一行一条语句。 
  • (、[ 之后，]、) 之前，不要添加任何空格。在 { 前后，在 } 之前添加空格。
  • 除个别情况外方法调用应当使用括号包裹参数，尤其是第一个参数以 ( 开头时，比如 f((3 + 2) + 1)；
  • 三元操作符的每个分支只写一个表达式。即不要嵌套三元操作符。嵌套情况请使用 if/else 结构。
  • ...
几个主要的《 Rails 风格指南》的影响：
  • 当需要为一个 RESTful 资源添加动作时（你真的需要吗？），应使用 member 路由和 collection 路由。
  • 当需要定义多个 member/collection 路由时，应使用块结构。
  • 控制器应该保持苗条 ― 它们应该只为视图层提供数据，不应包含任何业务逻辑（所有业务逻辑都应当放在模型里）。
  • 当只需要通过属性查询单个记录时，优先使用 find_by 而不是 where
  • 当需要处理多条记录时，应使用 find_each。
  • ...

ruby风格指南：
1.(、[ 之后，]、) 之前，不要添加任何空格。在 { 前后，在 } 之前添加空格。
2.避免在不需要的情况下使用 self。（只有在调用 self 的修改器、以保留字命名的方法、重载的运算符时才需要）
# 差
def ready?
  if self.last_reviewed_at > self.last_updated_at
    self.worker.update(self.content, self.options)
    self.status = :in_progress
  end
  self.status == :verified
end

# 好
def ready?
  if last_reviewed_at > last_updated_at
    worker.update(content, options)
    self.status = :in_progress
  end
  status == :verified
end
3.通过使用范围或 Comparable#between? 来替代复杂的比较逻辑。
# 差
do_something if x >= 1000 && x <= 2000

# 好
do_something if (1000..2000).include?(x)

# 好
do_something if x.between?(1000, 2000)
4.根据注解的不同标题标记不同类型的代码。

rails风格指南：
1.控制器应该保持苗条 ― 它们应该只为视图层提供数据，不应包含任何业务逻辑（所有业务逻辑都应当放在模型里）。
2.应使用 find_each 来迭代一系列 ActiveRecord 对象。用循环来处理数据库中的记录集（如 all 方法）是非常低效率的，因为循环试图一次性得到所有对象。而批处理方法允许一批批地处理记录，因此需要占用的内存大幅减少。
3.不要在查询中使用字符串插值，它会使你的代码有被 SQL 注入攻击的风险。
# 差——插值的参数不会被转义
Client.where("orders_count = #{params[:orders]}")

# 好——参数会被适当转义
Client.where('orders_count = ?', params[:orders])
4.当查询中有超过 1 个占位符时，应考虑使用名称占位符，而非位置占位符。
# 一般般
Client.where(
  'created_at >= ? AND created_at <= ?',
  params[:start_date], params[:end_date]
)

# 好
Client.where(
  'created_at >= :start_date AND created_at <= :end_date',
  start_date: params[:start_date], end_date: params[:end_date]
)
